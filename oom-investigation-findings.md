# OOM Killer Investigation Findings — kubevuln Subprocess SBOM

## Context

PR [#333](https://github.com/kubescape/kubevuln/pull/333) introduced subprocess-based SBOM creation using `moby/sys/reexec` + `os.Pipe` to isolate Syft SBOM generation in a child process. The goal: when the child is OOM-killed, the parent (kubevuln) survives and reports a structured error instead of the pod restarting.

**Result:** The pod still restarts on OOM. This document explains why and what the actual fix options are.

## PR #333 Architecture Summary

- **Reexec pattern** — same binary re-execs itself as `sbom-worker` via `reexec.Register("sbom-worker")` in `init()`
- **IPC via `os.Pipe` (FD 3/4)** — keeps stdout/stderr free for logging
- **Timeout enforcement** — goroutine + select with `context.Background()` (independent of HTTP request context)
- **Child tracking** — mutex-protected slice for SIGTERM propagation on shutdown
- **Sentinel errors** — `ErrChildOOMKilled` (SIGKILL), `ErrChildSignaled`, `ErrChildTimeout`
- **Feature flag** — `subprocessSBOM: false` by default, zero overhead when off
- **Graceful child death handling** — `handleChildError` detects SIGKILL and returns `ErrChildOOMKilled`; parent continues running

The code for detecting and handling child OOM death is correct. The problem is at the Linux/Kubernetes level.

## Root Cause: Two Blocking Issues

### Issue 1 (Critical): `memory.oom.group = 1` on the container cgroup

The container's cgroup v2 has `memory.oom.group` set to `1`:

```
Pod-level cgroup:       memory.oom.group = 0   (per-process selection)
Container-level cgroup: memory.oom.group = 1   (kills ALL processes together)
```

With `memory.oom.group = 1`, the kernel kills **every process in the cgroup as a group** during OOM. It does not evaluate individual `oom_score_adj` values or pick victims — all processes die together.

This is set by the kubelet/containerd. Modern Kubernetes with cgroup v2 sets `memory.oom.group = 1` at the container level by default. The `oom-killer-fix.md` document incorrectly assumed the default is `0`.

**Consequence:** No amount of `oom_score_adj` tuning will help. Parent and child share the same container cgroup, so both are killed together.

**Verified via debug pod:**

```
# Container cgroup memory.oom.group
nsenter -t 1 -m -- cat /sys/fs/cgroup/kubelet.slice/.../cri-containerd-<id>.scope/memory.oom.group
1

# Memory nearly maxed
memory.current: 536,412,160
memory.max:     536,870,912   (512Mi, 99.9% used)
```

### Issue 2: `CAP_SYS_RESOURCE` is not effective despite being in the pod spec

The deployment has `SYS_RESOURCE` in `capabilities.add`. The OCI runtime spec generated by containerd **does** include `SYS_RESOURCE` in the `bounding`, `effective`, and `permitted` sets — but **not** in `inheritable` or `ambient`.

For a non-root user (UID 65532), Linux capability transition rules during `execve` require the `ambient` set for capabilities to survive:

```
P'(effective) = P'(ambient) = 0    (ambient was never set → caps lost)
```

**Verified via debug pod with hostPID:**

```
CapInh: 0000000000000000   (empty)
CapPrm: 0000000000000000   (empty)
CapEff: 0000000000000000   (empty — SYS_RESOURCE not usable)
CapBnd: 00000000a90425fb   (SYS_RESOURCE present but unreachable)
CapAmb: 0000000000000000   (empty — this is why caps are lost)
```

**Runtime versions:** containerd v2.1.1, runc v1.3.0 — the OCI spec simply doesn't include `inheritable`/`ambient` sets for non-root containers.

Changing `allowPrivilegeEscalation` between `true`/`false` does not fix this — the root cause is the missing ambient set in the OCI spec.

## Why the Subprocess Approach Cannot Work in kubevuln (as-is)

1. `memory.oom.group = 1` means all processes in the container cgroup die together on OOM
2. Even if that were `0`, the process can't write `oom_score_adj` because the capability is ineffective
3. The cgroup directory is owned by root and not writable by the non-root process (UID 65532)

## Fix Options Evaluated

### Option A: `oom_score_adj` tuning (from `oom-killer-fix.md`)

**Status: Will not work**

- Blocked by `memory.oom.group = 1` (makes per-process OOM scoring irrelevant)
- Blocked by ineffective `CAP_SYS_RESOURCE` for non-root user
- Would require: fixing both issues above, which means running as root + changing `memory.oom.group`

### Option B: File capabilities on the binary (`setcap`)

**Status: Only solves capability issue, not `memory.oom.group`**

Adding `setcap cap_sys_resource=eip /kubevuln` in the Dockerfile would make the capability effective after exec. But `memory.oom.group = 1` still kills all processes together. Not sufficient alone.

### Option C: Child cgroup approach in kubevuln

**Status: Requires significant deployment changes**

Create a sub-cgroup for the worker process with its own memory limit. The sub-cgroup defaults to `memory.oom.group = 0`, so OOM kills only the worker.

Required deployment changes:
- `runAsUser: 0` (cgroup dirs are root-owned)
- `runAsNonRoot: false`
- `hostPath` volume mount for `/sys/fs/cgroup` (read-write)
- Code to discover cgroup path via `/proc/self/mountinfo`, create sub-cgroup, set `memory.max`, move child PID

**Trade-off:** Running kubevuln as root is a security regression.

### Option D: Move SBOM generation to node-agent

**Status: Most practical — almost everything already in place**

The node-agent DaemonSet is already highly privileged:

| Requirement | node-agent |
|---|---|
| `runAsUser: 0` (root) | ✅ Already set |
| `SYS_ADMIN` capability | ✅ Already set |
| `SYS_RESOURCE` capability | ✅ Already set |
| `hostPID: true` | ✅ Already set |
| `/sys/fs/cgroup` from host | ✅ Mounted (read-only) |
| AppArmor: Unconfined | ✅ Already set |

**One change needed:** Make the `/sys/fs/cgroup` volume mount read-write:

```yaml
# Current:
- mountPath: /sys/fs/cgroup
  name: cgroup
  readOnly: true     # ← change to false

# Fixed:
- mountPath: /sys/fs/cgroup
  name: cgroup
  readOnly: false
```

Since node-agent runs as root, all capabilities are in the effective set (no ambient cap issue). With `SYS_ADMIN`, the process can create sub-cgroups, set memory limits, and manage child processes.

The SBOM worker subprocess would run in a child cgroup with its own `memory.max` and `memory.oom.group = 0`, so OOM kills only the worker while node-agent survives.

### Option E: Sidecar container

**Status: Recommended — Kubernetes-native, portable, no special privileges**

Run the SBOM worker as a separate container in the same pod (kubevuln or node-agent). Kubernetes automatically gives each container its own cgroup with its own `memory.max`. When the sidecar OOMs, only that container restarts — the main container survives.

IPC between the main container and the sidecar can use a shared `emptyDir` volume (JSON files) or localhost networking.

**Trade-off:** More upfront architectural work (IPC, lifecycle coordination between containers).

## Recommendation

**Option E (sidecar container)** is the recommended path.

### Why not Option D (child cgroup in node-agent)?

While Option D is technically feasible (node-agent already has root + `SYS_ADMIN` + host cgroup mount), it operates at the Linux cgroup layer alongside the kubelet and container runtime — both of which assume they own the cgroup hierarchy. This creates significant risks for a product shipping to diverse customer environments:

- **cgroup path format varies** across cgroup drivers (systemd vs cgroupfs), container runtimes, and Kubernetes distributions (EKS, GKE, AKS, OpenShift, RKE, k3s). Discovery logic would need to handle all variants.
- **cgroup v1 vs v2** — some customer clusters still run cgroup v1, which has a completely different API (separate per-controller hierarchies, no `memory.oom.group`). The code would need to support both.
- **Race conditions with kubelet** — the kubelet periodically reconciles cgroup settings. Unexpected sub-cgroups may cause undefined behavior or be cleaned up.
- **Security policy conflicts** — hardened clusters (OPA/Gatekeeper, Kyverno, PodSecurity) may reject the writable `/sys/fs/cgroup` hostPath mount or flag it in audits.
- **Debugging burden** — failures would require investigating the intersection of kernel version, cgroup driver, runtime version, kubelet config, and SELinux/AppArmor policies. This is the kind of deep debugging that doesn't scale across a customer base.

### Why Option E (sidecar container)?

- **Kubernetes-native** — uses standard pod spec primitives, works on every conformant cluster regardless of runtime, cgroup version, or distribution.
- **Cgroup isolation for free** — Kubernetes gives each container its own cgroup with its own `memory.max`. No manual cgroup manipulation needed.
- **No special privileges** — no root, no `SYS_ADMIN`, no `hostPath` mounts, no cgroup filesystem access. Works with strict PodSecurity policies.
- **Portable** — identical behavior on cgroup v1 and v2, every runtime, every distribution.
- **Observable with standard tooling** — `kubectl describe pod` shows the sidecar's OOM kill, restart count, and memory usage. No need for host-level debugging.
- **Clear responsibility boundary** — Kubernetes manages the cgroup lifecycle; the application only manages the IPC and work coordination.

The trade-off is more upfront engineering (IPC protocol, container lifecycle coordination, Helm chart changes), but the operational surface area in production is dramatically smaller than managing cgroups directly.
